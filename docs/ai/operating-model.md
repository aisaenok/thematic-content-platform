# Operating model для AI-ассистентов

Этот документ описывает правила использования AI-ассистентов и Codex в проекте **Thematic Content Platform**.

Цель — ускорить разработку, не потеряв контроль над архитектурой.

## Назначение проекта

**Thematic Content Platform** — портфолио-прототип wiki/content-платформы для тематических энциклопедических сайтов.

Проект создается как greenfield-модернизация старого Alien Wiki. Старый проект используется как источник идеи, доменной области и набора контентных сущностей, но новая версия не является прямой миграцией старого кода.

## Главная идея AI-подхода

AI должен помогать выполнять ограниченные инженерные задачи, а не самостоятельно проектировать всю систему.

Хорошая задача для AI:

```text
Создай типы для article-сущности в packages/content-domain.
```

Плохая задача для AI:

```text
Сделай всю content platform на Next.js.
```

## Что можно поручать AI

### 1. Создание локального кода

Можно поручать:

- создание TypeScript-типов;
- создание простых компонентов;
- создание mock data;
- реализацию маленьких утилит;
- добавление barrel exports;
- подготовку простых тестов;
- создание небольшой страницы;
- добавление README-секций;
- генерацию документации по уже принятому решению.

Примеры:

```text
Создай ArticleCard-компонент.
```

```text
Добавь mock content source с getArticles и getArticleBySlug.
```

```text
Создай страницу /wiki/[slug] на основе существующего content adapter.
```

### 2. Локальные исправления

Можно поручать:

- исправить TypeScript-ошибку в конкретном файле;
- привести импорт к архитектурному правилу;
- упростить компонент;
- убрать дублирование в пределах одного модуля;
- добавить обработку пустого состояния.

### 3. Документация

Можно поручать:

- обновить README;
- описать архитектурное решение после того, как оно принято человеком;
- оформить task brief;
- подготовить modernization notes;
- добавить комментарии к неочевидной domain-логике.

## Что нельзя поручать AI без отдельного решения

AI не должен сам:

- выбирать CMS;
- добавлять backend;
- добавлять auth;
- добавлять Redux, MobX, Redux-Saga или другой глобальный state manager;
- менять routing strategy;
- менять структуру монорепозитория;
- переписывать проект на другой стек;
- добавлять UI kit;
- добавлять Docker;
- добавлять CI/CD;
- вводить FSD или менять слои без отдельного решения;
- делать большой cross-cutting refactoring.

Если такая задача нужна, сначала должно быть принято архитектурное решение, а потом Codex получает маленькие исполнительские задачи.

## Принятые ограничения

### 1. Не добавлять Redux-Saga

В старом Alien Wiki была Redux-Saga, но в новой версии она считается избыточной.

Причина:

- проект content-driven;
- основная работа — чтение контента;
- сложных бизнес-процессов и оркестрации пока нет;
- Next.js server layer лучше подходит для загрузки данных;
- для клиентских интерактивных частей можно рассмотреть React Query/SWR;
- для UI-state можно использовать local state, Context или Zustand, если появится реальная необходимость.

### 2. Не добавлять backend на первом этапе

На первом этапе backend не нужен.

Сначала нужно доказать вертикальный срез:

```text
typed content model → mock content source → Next page → UI → SEO metadata
```

Отдельный backend можно обсуждать позже, если появятся реальные причины:

- сложный search;
- отдельный public API;
- сложная CMS-интеграция;
- авторизация;
- админка;
- фоновые процессы;
- интеграции, которые не хочется держать в Next server layer.

### 3. Не добавлять CMS на первом этапе

На первом этапе достаточно mock/local content source.

CMS можно добавить позже через adapter layer, чтобы не завязать доменную модель на конкретную CMS.

Потенциальные варианты CMS:

- Strapi;
- Payload;
- Directus;
- Sanity;
- собственный простой data source.

Выбор CMS — отдельное архитектурное решение.

### 4. Не смешивать domain и UI

Domain-типы не должны зависеть от React, Next.js, CSS, компонентов или UI kit.

Пример правильного направления зависимости:

```text
apps/web → packages/content-source → packages/content-domain
```

Неправильно:

```text
packages/content-domain → apps/web
packages/content-domain → React components
packages/content-domain → Next.js types
```

### 5. Не усложнять раньше времени

Проект должен выглядеть как аккуратный portfolio-grade prototype, а не как недостроенный enterprise-комбайн.

На старте не нужны:

- role model;
- permissions;
- admin panel;
- complex workflow;
- event bus;
- microservices;
- queues;
- Saga;
- custom SSR;
- самописный framework.

## Definition of Done для AI-задачи

Задача считается готовой, если:

- изменены только явно указанные области;
- не добавлены лишние зависимости;
- не изменена архитектура без запроса;
- код типизирован;
- публичные типы экспортированы через `index.ts`, если это пакет;
- нет смешения domain/UI;
- есть понятные имена;
- нет временных `console.log`;
- нет TODO без причины;
- проект проходит доступные проверки.

Минимальные проверки, когда они появятся:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

Если команд еще нет, AI должен явно указать, что проверить автоматически пока нечем.

## Как формулировать задачи для Codex

### Хороший формат

```text
Задача:
Создать mock content source для статей.

Контекст:
Проект — Nx + Next.js portfolio-grade content platform.
Сейчас CMS нет, данные берутся из local mock adapter.

Файлы:
- packages/content-source/src/mock/articles.ts
- packages/content-source/src/index.ts

Ограничения:
- не добавлять CMS;
- не добавлять backend;
- не добавлять React-компоненты;
- не менять apps/web;
- не добавлять runtime-зависимости.

Ожидаемый результат:
- getArticles();
- getArticleBySlug(slug);
- getRelatedArticles(articleId);
- статичные данные для 3-5 статей;
- экспорт из index.ts.
```

### Плохой формат

```text
Добавь статьи.
```

Проблема: непонятно, где, в каком слое, с какой моделью, с какими ограничениями.

## Как ревьюить результат AI

После генерации нужно проверить:

- не добавились ли лишние пакеты;
- не появились ли лишние слои;
- не появились ли прямые зависимости domain от UI;
- не сделал ли AI больше, чем просили;
- не использовал ли устаревшие API;
- не добавил ли искусственную сложность;
- не нарушил ли naming;
- можно ли объяснить решение на собеседовании.

Главный критерий:

> Я должен быть способен сам объяснить каждую архитектурную границу и каждую зависимость в проекте.

Если код выглядит “магически” или ты не можешь его объяснить, его нельзя оставлять как есть.

## Когда останавливать AI

Остановить Codex и переписать задачу нужно, если он:

- начинает менять много файлов вне scope;
- предлагает добавить Redux/Saga/backend/CMS без запроса;
- делает большой refactoring вместо локального изменения;
- смешивает domain и UI;
- добавляет абстракции без необходимости;
- пишет слишком общий framework вместо конкретной задачи;
- не может объяснить, почему решение нужно.

## Рабочая стратегия

Разработка должна идти маленькими вертикальными срезами:

```text
1. Domain model
2. Mock data source
3. Server page
4. UI block
5. SEO metadata
6. Tests/docs
```

Не нужно сначала строить всю платформу. Лучше сделать маленький, но завершенный сценарий, который можно показать как архитектурный пример.
